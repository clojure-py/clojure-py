# Type System Decisions

The type system for the clojure-py runtime substrate. Decisions made during
the initial-runtime design round. Companion to
`dispatch-architecture-summary.md` and `rc-research-findings.md`.

## Core shape

- **Fat 16-byte Value, passed by copy.** Every value in the runtime is a
  `repr(C, align(16))` `Value { tag: u32, _pad: u32, payload: u64 }`.
  Stack and arg-list traffic is `&[Value]` — borrowed slices, no
  per-call allocations.
- **Tag IS the type_id.** One unified `u32` namespace covers primitives
  and heap types. Dispatch always reads `value.tag` directly; there is
  no separate "is this heap?" branch on the hot path. The RC fast-path
  skip is a single compare (`tag >= FIRST_HEAP_TYPE`).
- **Native heap, Python objects are foreign.** Heap-allocated Clojure
  values live in our own RC-managed heap. Python objects are held
  through a reserved `TYPE_PYOBJECT` tag whose payload is an opaque
  `*mut PyObject`; the kernel never dereferences it. The PyO3 binding
  crate (later) is the only thing that knows how to incref/decref a
  pyobject-tagged Value.

## TypeId space

```
TYPE_NIL       = 0
TYPE_BOOL      = 1
TYPE_INT64     = 2     // full i64 range; not a fixnum
TYPE_FLOAT64   = 3     // f64::to_bits
TYPE_CHAR      = 4     // Unicode scalar in low 32 bits
TYPE_PYOBJECT  = 5     // reserved for future PyO3 binding
// 6..15 reserved for future primitives
FIRST_HEAP_TYPE = 16
```

- u32 type IDs are interned in a global registry. The first 16 are
  fixed at compile time; heap type IDs are allocated dynamically from
  16 upward.
- Type IDs are **not stable across runs**. Inventory iteration order
  is unspecified, so static registrations may receive different IDs in
  different builds. Clients always go through a `OnceCell<TypeId>`
  populated at init.
- v1 capacity: 65,536 distinct types. A second-level slot table can be
  added later if needed.

## Heap object header (16 bytes, aligned 16)

```rust
#[repr(C, align(16))]
pub struct Header {
    pub type_id: TypeId,    // 4 bytes — must match the Value.tag pointing here
    pub flags:   u32,       // 4 bytes — reserved (transient, frozen, future use)
    pub rc:      AtomicI32, // 4 bytes — Lean-4-style signed counter; see below
    _pad:        u32,       // 4 bytes — reserved
}
```

Body data follows the header. The redundant `type_id` lets recovery
paths (destructor, debugger) read the type without going through a
Value; debug builds assert it matches the pointing Value's tag.

## Reference counting

Single signed counter (`rc: AtomicI32`), Lean-4 / Perceus style:

- `rc < 0`: **biased mode**, magnitude is the live count, mutated
  non-atomically on the owner thread.
- `rc > 0`: **shared mode**, mutated atomically by any thread.
- `rc == 0`: drop-to-zero, ready for destruction.

Initial state on alloc: `rc = -1` (biased, count = 1). Cross-thread
sharing happens through a one-time `gc::share` op (CAS `-n → +n` with
Release ordering); thereafter the object is forever in shared mode.

## Calling convention

```rust
type MethodFn = unsafe extern "C" fn(args: *const Value, n: usize) -> Value;
// In Rust: fn(args: &[Value]) -> Value
```

- Receiver is `args[0]` by convention.
- Variadic / multi-arity is handled by `args.len()`, not by signature
  variants. Every method fn pointer in the IC has the same shape — a
  hard requirement for tier-1 dispatch performance.
- Returns a Value by copy. RC ownership of the return is "donated" to
  the caller (caller decides to retain via `gc::dup` or drop).
- Borrowed args (`&[Value]`) do **not** dup/drop on entry/exit. A
  callee retaining a value past return must `gc::dup` it explicitly.
  v1 leaves dup/drop insertion manual; Perceus-style automatic
  insertion is deferred (see `deferred-work.md`).

## Per-type metadata

```rust
pub struct TypeMeta {
    pub type_id:  TypeId,
    pub name:     &'static str,
    pub layout:   Layout,                  // body layout, beyond the Header
    pub destruct: unsafe fn(*mut Header),  // drops outgoing Value fields
    pub table:    ArcSwap<PerTypeTable>,   // tier-2 perfect-hash dispatch table
}
```

- One `TypeMeta` per type, leaked to `'static` (never freed; tiny
  amount of memory).
- Lookups by ID are an atomic load from a fixed-capacity slot array.
  No locks on the hot path.
- The destructor is generated by the `register_type!` macro: for each
  declared `Value` field, it emits `gc::drop`; non-Value fields are
  dropped by their Rust `Drop` or skipped if `Copy`.

## Open-world extension

- Static types, protocols, and impls register via `inventory::submit!`
  entries emitted by the proc-macros. `clojure_rt::init()` walks the
  inventory at startup and assigns IDs.
- Dynamic types (e.g. runtime `defrecord`) call
  `register_dynamic_type` directly; same code path as static.
- Adding a new protocol impl rebuilds the affected type's
  `PerTypeTable` (perfect-hash regenerated) and bumps the protocol
  method's `version` counter; all live IC slots invalidate lazily.
